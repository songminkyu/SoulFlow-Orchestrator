# Reviewer Execution Protocol

## 1. 변경 파악

- 변경 파일 목록 확인
- diff 분석
- 스펙 대비 변경 범위 확인

## 2. 다중 렌즈 검토

세 관점에서 순서대로 코드를 검토:

| 렌즈 | 핵심 질문 |
|------|-----------|
| **보안 전문가** | 이 코드가 공격자에게 어떻게 악용될 수 있는가? (주입/XSS/인증/권한) |
| **신규 입사자** | 코드를 처음 보는 사람이 동작을 오해할 수 있는 부분은? |
| **미래 유지보수자** | 6개월 후 이 코드가 버그 또는 의도치 않은 변경의 온상이 될 부분은? |

추가 검토 항목:
- 정확성: 로직이 스펙과 일치하는가
- 성능: 불필요한 연산, 메모리 누수
- 복잡도: YAGNI/DRY/KISS 위반

## 3. 이슈 분류

| 심각도 | 정의 | 행동 |
|--------|------|------|
| CRITICAL | 보안/데이터 손실 위험 | 즉시 수정 필수 |
| HIGH | 기능 오류 | 수정 권장 |
| MEDIUM | 개선 사항 | 선택적 수정 |
| LOW | 스타일/가독성 | 참고 |

## 3.5. 보고 전 자기 점검

보고하기 전 다음 질문에 스스로 답하라:

- [ ] 세 렌즈(보안/신규입사자/유지보수자) 모두 적용했는가?
- [ ] CRITICAL 이슈를 놓쳤을 가능성을 보안 전문가 관점에서 재확인했는가?
- [ ] 이슈마다 코드 원문 인용과 심각도 분류가 있는가?
- [ ] 칭찬할 부분도 최소 1건 포함했는가? (건설적 피드백의 기준)

하나라도 "아니오"이면 해당 렌즈 또는 항목으로 돌아간다.
