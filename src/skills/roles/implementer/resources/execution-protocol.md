# Implementer Execution Protocol

## 1. 스펙 확인

- 변경 대상 파일 목록 확인
- 요구사항 파악
- 불확실성 분류 (clarification-protocol)

## 2. 코드 탐색

- 기존 코드 패턴/컨벤션 파악
- 관련 모듈 의존성 확인

## 2.5. 설계 숙고 (복잡한 변경 시)

구현 방식이 2개 이상이거나 설계 결정이 포함된 경우 [spp-deliberation.md](../../_shared/spp-deliberation.md) 절차 실행:

- 접근법 후보를 2-3개 나열
- 회의론자 관점에서 각 접근법의 약점 식별
- 선택 + 트레이드오프 한 줄 기록 후 구현 시작

## 3. 구현

- 스펙 기준 파일 수정
- 기존 패턴 준수
- 최소 변경 원칙
- 언어별 컨벤션 적용 (`_shared/lang/` 참조)

## 4. 셀프 검증

- 빌드 확인 (언어별 빌드 명령 → `_shared/lang/` 참조)
- 기본 동작 테스트
- 변경 범위가 스펙과 일치하는지 확인

## 4.5. 보고 전 자기 점검

보고하기 전 다음 질문에 스스로 답하라:

- [ ] 스펙에서 요구한 모든 항목이 구현되었는가?
- [ ] 설계 숙고가 필요한 결정을 프로토콜 없이 직감으로 내리지 않았는가?
- [ ] 변경 파일 외의 파일을 의도치 않게 수정하지 않았는가?
- [ ] 빌드/테스트가 실제로 통과했는가 (가정이 아닌가)?

하나라도 "아니오"이면 해당 단계로 돌아간다.

## 5. 보고

- 변경 파일 목록
- 셀프 검증 결과
- 미해결 이슈 (있으면)

## 에러 처리

- error-escalation 규칙 적용
- [에러 복구 시나리오](error-playbook.md) 참조
