# Debugger Execution Protocol

## 1. 증상 파악

- 에러 메시지/로그 확인
- 재현 조건 파악
- 영향 범위 추정

## 2. 범위 축소

- 관련 파일 탐색
- 호출 경로 추적
- 이분 탐색으로 원인 위치 특정

## 2.5. 가설 생성 (RCA 전 필수)

가능한 원인을 **3개 이상** 먼저 나열한 후, 각 가설을 반증하는 테스트를 설계:

```
가설 A: [원인 추정] — 근거: [증거]
가설 B: [원인 추정] — 근거: [증거]
가설 C: [원인 추정] — 근거: [증거]

반증 테스트:
- A 반증: [확인 방법] → [결과]
- B 반증: [확인 방법] → [결과]
남은 가설 → RCA 대상
```

가설 없이 코드 수정 금지. 반증되지 않은 가설은 근본 원인이 아님.

## 3. 근본 원인 분석 (RCA)

- 직접 원인 vs 근본 원인 구분
- 코드 흐름 추적
- 데이터 상태 확인

## 3.5. 보고 전 자기 점검

보고하기 전 다음 질문에 스스로 답하라:

- [ ] 최소 3개 가설을 세우고 각각 반증을 시도했는가?
- [ ] 남은 가설이 하나뿐이거나, 남은 가설들 중 RCA 대상을 명시했는가?
- [ ] "아마 이것 같다"가 아니라 증거로 원인을 확정했는가?
- [ ] 수정 제안이 직접 원인이 아닌 근본 원인을 겨냥하는가?

하나라도 "아니오"이면 2.5 가설 생성 단계로 돌아간다.

## 4. 보고

```
**증상**: [에러 메시지/동작]
**재현**: [재현 조건]
**원인**: [근본 원인]
**수정 제안**: [구체적 코드 변경]
**영향**: [수정 시 영향 범위]
```

## 금지 사항

- 원인 미확인 상태에서 코드 수정
- 증상만 보고 추측 수정
