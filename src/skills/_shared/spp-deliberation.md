# SPP 숙고 프로토콜

복잡한 결정 전 단일 모델이 내부적으로 다각 관점을 시뮬레이션하는 절차.

## 적용 조건

다음 중 하나라도 해당하면 이 프로토콜을 실행:

- 구현 방식이 2개 이상 존재
- 부작용 또는 되돌리기 어려운 변경 포함
- 근본 원인이 즉시 명확하지 않은 문제
- 설계 결정이 미래 유지보수에 영향

## 숙고 절차

### 1단계 — 관점 수집

응답하기 전 세 관점에서 각 1-2문장으로 검토:

| 관점 | 핵심 질문 |
|------|----------|
| **실용주의자** | 가장 단순하게 작동하는 해결책은? |
| **회의론자** | 이 접근법의 가장 큰 약점 또는 실패 조건은? |
| **유지보수자** | 6개월 후 이 코드/결정을 처음 보는 사람이 혼란스러울 부분은? |

### 2단계 — 선택 기록

숙고 결과를 한 줄로 기록 (사용자에게 보여도 좋음):

```
선택: [접근법 A]
이유: [실용주의자 관점 근거]
트레이드오프: [회의론자가 지적한 약점] → [수용/완화 방법]
```

### 3단계 — Pre-mortem

"이 접근법이 실패한다면, 가장 가능성 높은 이유 1가지는?"
→ 실패 시나리오를 명시하고, 대비책을 포함하거나 사용자에게 알림.

## 예시 (Few-shot)

> 상황: 세션 만료 시 메모리를 자동 보존하는 방법을 선택해야 함.

**1단계 — 관점 수집**

| 관점 | 검토 |
|------|------|
| **실용주의자** | 기존 `consolidation` 훅에 세션 순회 로직 추가. 새 컴포넌트 없이 동작. |
| **회의론자** | 세션 수가 많으면 consolidation이 느려진다. N개 세션 × LLM 호출 = 비용 폭발. |
| **유지보수자** | consolidation 안에 세션 로직이 섞이면 6개월 뒤 두 책임이 뒤엉켜 수정 범위가 불명확해진다. |

**2단계 — 선택 기록**

```
선택: consolidation 훅에 결정론적 추출 추가 (LLM 미사용)
이유: 기존 흐름 재사용, 새 의존성 없음
트레이드오프: consolidation 책임 증가 → promote_sessions_to_daily()로 함수 분리해 응집도 유지
```

**3단계 — Pre-mortem**

실패 시나리오: 세션 스토어가 `list_by_prefix`를 구현하지 않으면 승격이 조용히 건너뜀.
대비책: `if (!sessions.list_by_prefix) return { promoted: 0, skipped: 0 }` — 명시적 early return.

---

## 참고

- 간단한 작업(명확한 단일 해결책)에는 적용하지 않음
- 각 단계는 출력 형식의 길이를 최소화
- "어떤 것이 더 나은가"가 아니라 "이것이 실패하는 경우"에 초점
