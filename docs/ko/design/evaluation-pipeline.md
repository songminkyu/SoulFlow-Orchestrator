# 평가 파이프라인 설계

## 목적

평가 파이프라인은 동일한 시나리오를 직접, 모델 직접, 워크플로우, 에이전트 경로에서
**로컬 우선, 프로바이더 중립** 방식으로 실행하고
결과를 시간에 따라 비교할 수 있는 기반을 제공한다.

유닛 테스트나 통합 테스트를 대체하는 것이 목적이 아니다.
다음과 같은 질문에 답하기 위한 것이다:

- `claude_sdk`에서 `ollama`로 전환하면 이 사용 사례의 답변 품질이 달라지는가?
- 이번 프롬프트 변경이 라우팅 정확도를 개선했는가, 저하시켰는가?
- 이 카테고리의 요청에 어떤 실행 경로가 가장 빠른가?

## 핵심 개념

### EvalCase

`EvalCase`는 단일 테스트 시나리오다. 다음을 포함한다:

- 입력 메시지
- 예상 실행 경로 (직접 / 모델 / 에이전트)
- 예상 출력 형태 또는 내용 기준
- 번들 그룹화를 위한 선택적 태그

케이스는 버전 관리되며 JSON 픽스처로 저장된다.
특정 프로바이더가 사용 가능한지 여부에 의존하지 않는다.

### EvalDataset

도메인이나 목적으로 그룹화된 `EvalCase` 모음.
태그로 필터링하여 집중적인 서브셋을 실행할 수 있다.

### EvalRunner

러너는 케이스를 라이브 시스템에 대해 재생한다.
다음을 캡처한다:

- 실제 실행된 경로
- 원시 출력 내용
- 지연 시간과 토큰 비용
- 중단 이유

러너는 결과를 판단하지 않는다 — 캡처만 한다.

### Judge / Scorer

Judge는 러너와 독립적으로 캡처된 출력을 평가한다.

결정론적 judge는 다음을 확인한다:

- 올바른 실행 경로가 실행됐는가?
- 출력이 예상 스키마와 일치하는가?
- 출력에 필수 필드가 포함됐는가?

선택적인 `LlmJudgePort`는 개방형 품질 검사를 모델에 위임할 수 있지만,
결정론적 judge가 항상 먼저 실행된다.
LLM judge가 유일한 judge가 되는 경우는 없다.

### Scorecard와 RunReport

`Scorecard`는 하나의 평가 실행을 요약한다:
라우팅 정확도, 스키마 적합성, 품질 통과율, 평균 지연 시간, 비용.

`RunReport`는 원시 출력과 케이스별 judge 결과를 포함한 전체 실행 기록을 저장한다.
보고서는 베이스라인과 비교하여 회귀를 감지할 수 있다.

## CLI 진입점

```
npm run eval:smoke    # 빠른 서브셋 — 핵심 라우팅과 스키마 검사만
npm run eval:full     # 전체 번들 — 품질, 지연 시간, 비용 전체 비교
```

`eval:smoke`는 CI 게이트에 적합하다.
`eval:full`은 릴리스 전 검증이나 설정 변경 후 검토에 사용한다.

## 일반 테스트와의 관계

평가 실행은 회귀 테스트를 대체하지 않는다.

| | 테스트 | 평가 실행 |
|---|---|---|
| 목적 | 코드 정확성 | LLM 동작 품질 |
| 픽스처 유형 | 코드 어설션 | 시나리오 + 판단 |
| CI 역할 | 필수 게이트 | 선택적 게이트 |
| 출력 | 통과 / 실패 | 스코어카드 + diff |

## 프로바이더 비교

러너가 프로바이더 중립이기 때문에, 동일한 `EvalDataset`을
서로 다른 백엔드 설정에 대해 실행할 수 있다.

이것이 클라우드 프로바이더에서 로컬 모델(Ollama)로 전환했을 때
특정 워크로드의 출력 품질이 저하되는지 평가하는 주요 사용 사례다.
