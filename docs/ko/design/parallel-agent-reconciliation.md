# 병렬 에이전트 조정(Reconciliation) 설계

## 목적

Phase Loop가 여러 병렬 브랜치로 분기하면, 각 브랜치는 독립적인 결과를 생성한다.
조정(Reconciliation)은 워크플로우가 다음 단계로 진행하기 전에
이 결과들을 하나의 일관된 출력으로 수렴시키는 단계다.

설계 목표는 **충돌 감지 → 통합 → 제한된 비평** 이다 —
자유로운 논쟁이 아니다.

## 조정이 적용되는 경우

다음 조건에서 조정이 활성화된다:

- 워크플로우 페이즈가 두 개 이상의 병렬 에이전트를 실행하는 경우
- 에이전트 출력이 사실, 결정 또는 권장 사항에서 서로 불일치하는 경우
- 워크플로우가 계속 진행하기 위해 하나의 결과가 필요한 경우

조정 단계가 없으면 워크플로우는 임의로 하나의 브랜치를 선택하거나
혼란스러운 다중 답변을 사용자에게 노출하게 된다.

## 조정 노드 (ReconcileNode)

`ReconcileNode`는 병렬 워크플로우 페이즈의 fan-in 지점에 위치한다.

모든 병렬 브랜치의 출력을 담은 `ParallelResultEnvelope`를 받아
하나의 병합된 결과를 생성한다.

조정 노드는 새로운 내용을 만들지 않는다.
브랜치가 이미 반환한 내용을 기반으로 선택, 병합, 또는 비평 요청을 수행한다.

## 충돌 감지

모델이 개입하기 전에, 조정 노드가 충돌을 결정론적으로 해결할 수 있는지 확인한다:

- 하나의 브랜치만 성공했으면 그 결과를 선택한다
- 브랜치들이 동의하면 합의된 결과를 통과시킨다
- 알려진 규칙이 있는 구조화된 필드에서 불일치가 발생하면 규칙을 적용한다 (예: 신뢰도 높은 것 우선)
- 해결 불가능한 불일치만 비평자에게 에스컬레이션한다

결정론적으로 해결 가능한 충돌을 모델 중재자에게 보내는 것은
이 시스템이 명시적으로 피하는 설계 실수다.

## 병합 전략

| 전략 | 사용 시점 |
|------|-----------|
| `select` | 하나의 브랜치 결과를 선택; 나머지는 폐기 |
| `merge` | 호환되는 출력을 하나로 결합 |
| `escalate` | 불일치를 해결할 수 없음; 사람 또는 비평자 검토 필요 |

## 비평 게이트 (CriticGate)

에스컬레이션이 발생하면 `CriticGate`가 충돌 요약을 받아
하나 이상의 브랜치에 수정을 요청할 수 있다.

비평자는 `RetryBudget` 하에서 작동한다.
기본값: 1회 재시도 허용, 최대 2회.
예산이 소진되면 워크플로우는 해결되지 않은 충돌을 기록하고
최선의 가용 결과로 계속하거나 사용자에게 에스컬레이션한다.

비평자는 브랜치를 무제한으로 재실행할 권한이 없다.
이 제한이 조정을 논쟁 루프와 구별짓는 핵심이다.

## 워크플로우 스키마

Phase Loop 워크플로우는 `fanout`과 `reconcile` 단계를
일급 스키마 요소로 포함한다.
조정 단계는 업스트림 브랜치를 참조하고
병합 전략과 재시도 예산을 지정한다.

이를 통해 조정 동작이 일반 노드 로직 안에 숨겨지지 않고
워크플로우 그래프 에디터에서 직접 확인할 수 있다.

## 가시성

조정 결과, 충돌 집합, 비평 결정은 워크플로우 실행 기록과 함께 저장된다.
워크플로우 상세 뷰는 다음을 표시한다:

- 어떤 브랜치가 동의하거나 불일치했는지
- 어떤 전략이 적용됐는지
- 비평 재시도가 트리거됐는지와 그 결과
