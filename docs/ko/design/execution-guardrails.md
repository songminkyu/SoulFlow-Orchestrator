# 실행 가드레일 설계

## 목적

실행 가드레일은 `once / agent / task / workflow` 실행 경로 전반에 걸쳐
**제한된 실행 정책**을 공유하는 계층이다.

두 가지 주요 실패 패턴을 방지하는 것이 목표다:

- 에이전트가 이미 찾아낸 정보를 도구 호출로 다시 탐색하는 낭비
- 하드 예산 없이 도구 호출이 무한 반복되는 상황

프로바이더 스로틀링이나 UI 출력 조절이 목적이 아니다.
실행 계층이 자신이 이미 알고 있는 것을 인식하고,
충분한 자원을 사용했을 때 멈추도록 하는 것이 목적이다.

## 두 계층

가드레일은 소프트 제약과 하드 제약을 의도적으로 분리한다.

**소프트 게이트 — 세션 재사용 및 신선도:**

도구 호출을 실행하기 전, 런타임이 다음을 확인한다:

- 최근 세션 턴에 이미 답변된 내용이 있는가?
- 현재 쿼리가 최근 검색과 동일하거나 유사한가?
- 기존 증거가 아직 충분히 신선한가?

조건을 만족하면 도구 호출을 건너뛰고 기존 증거를 재사용한다.
이전 시도가 실패하거나 미완성인 경우 재사용이 차단된다 —
실패한 세션은 신뢰할 수 있는 증거로 취급하지 않는다.
사용자가 명시적으로 재시도를 요청하면(`다시`, `retry`, `try again`)
재사용 게이트를 우회하고 새로운 실행을 시작한다.

**하드 게이트 — 도구 호출 예산:**

`orchestration.maxToolCallsPerRun`이 런당 도구 호출 상한을 설정한다.
상한에 도달하면 `max_tool_calls_exceeded` 중단 이유와 함께 실행이 종료된다.
once, agent, task, 워크플로우 fan-out 등 모든 실행 경로에 동일하게 적용된다.
값을 `0`으로 설정하면 제한을 비활성화한다.

## 신선도 윈도우

세션 증거는 만료된다.
이전 턴이 `orchestration.freshnessWindowMs`보다 오래된 경우,
토픽이 일치하더라도 캐시를 만료된 것으로 간주하고 새로운 검색을 실행한다.

시간에 민감한 쿼리는 짧은 신선도 윈도우로 재확인을 강제하고,
안정적인 도메인의 답변은 긴 윈도우로 신뢰 있게 재사용할 수 있다.

## 실패 인식 재사용

이전 세션 시도는 결과에 따라 분류된다:

- 성공 / 완료 → 재사용 가능
- 실패 / 취소 / 미완성 → 재사용 제외

이 구분 덕분에 런타임이 실패한 이전 실행을 유효한 답변으로 요약하는 오류를 방지한다.

## 중단 이유

가드레일이 실행을 종료할 때 중단 이유가 명시적으로 기록된다:

| 중단 이유 | 트리거 |
|-----------|--------|
| `reuse_summary` | 기존 세션 증거로 답변 |
| `same_topic` | 신선도 윈도우 내 중복 쿼리 |
| `stale_retry` | 증거가 너무 오래되어 새로운 검색 필요 |
| `retry_failed_attempt` | 이전 시도 실패 — 명시적 재시도 경로 진입 |
| `new_search` | 이전 증거 없음 — 정상 새 실행 |
| `max_tool_calls_exceeded` | 하드 예산 초과 |

이 이유들은 실행 로그에 기록되며 대시보드에서 운영자가 확인할 수 있다.

## 턴 제한과의 관계

`agentLoopMaxTurns`와 `taskLoopMaxTurns`는 추론 턴을 제한한다.
`maxToolCallsPerRun`은 턴 수와 무관하게 도구 실행 횟수를 제한한다.

이 둘은 서로를 대체할 수 없는 별개의 제약이다.
도구 호출 없이 긴 추론 체인이 이어져도 도구 예산은 소비되지 않는다.
단일 턴에서 여러 도구가 실행되면 그 횟수만큼 예산이 차감된다.
