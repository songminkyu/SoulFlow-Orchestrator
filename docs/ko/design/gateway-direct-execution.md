# 게이트웨이 / 직접 실행 설계

## 목적

실행 게이트웨이는 들어오는 요청을 조기에 분류하고
실제로 처리 가능한 가장 저렴한 경로로 라우팅한다.

세 가지 경로는 다음과 같다:

- **no-token** — 슬래시 커맨드, 상태 조회, 가드 결정 — LLM 불필요
- **model-direct** — 단일 턴 생성, 지속적인 도구 루프 불필요
- **agent** — 다단계 추론, 도구 호출, 세션 연속성 필요

이 경로들을 분리하면 불필요한 에이전트 호출이 줄어들고
단순 요청의 실행 지연 시간이 예측 가능해진다.

## 분류

각 인바운드 메시지는 다음을 기록하는 `RequestPlan`을 생성한다:

- 선택된 실행 경로
- 선택 근거 (분류 증거)
- 처리할 프로바이더와 실행기
- 결과를 반환할 채널

분류는 프로바이더가 호출되기 전에 이루어진다.
분류기는 요청 내용, 현재 세션 상태, 사용 가능한 도구, 활성 워크플로우 컨텍스트를 고려한다.

## 실행 경로

### 직접 도구 경로

일부 요청은 도구나 액션을 명시적으로 지정한다.
게이트웨이는 에이전트 루프를 구성하지 않고 직접 실행할 수 있다.
결과는 원래 채널로 즉시 반환된다.

예시: `/secret list`, `/cron status`, 승인 대기, 데이터베이스 조회.

### 모델 직접 경로

언어 모델 출력이 필요하지만 지속적인 도구 사용이 불필요한 요청은
단일 추론 호출로 처리된다.
세션이 유지되지 않고 도구 루프가 실행되지 않는다.

예시: 단순 설명, 형식 변환, 일회성 요약.

### 에이전트 경로

도구 사용, 다중 턴 추론, 세션 메모리가 필요한 요청은
전체 에이전트 루프로 전달된다.
프로바이더 기능과 CircuitBreaker 상태를 기반으로 에이전트 백엔드가 선택된다.

## 결과 봉투 (Result Envelope)

모든 경로는 일관된 형태의 `ResultEnvelope`를 생성한다:

- 원본 요청 참조
- 실행된 경로
- 응답 내용
- 중단 이유
- 비용 및 지연 시간 메타데이터

이를 통해 직접 경로와 에이전트 경로의 결과를 동일한 스코어카드에서 비교할 수 있고,
어느 경로가 실행됐든 대시보드 렌더링이 일관성을 유지한다.

## 채널 어피니티

결과는 항상 요청이 발생한 채널로 반환된다.
워크플로우 페이즈가 Slack 스레드용 출력을 생성하면
그 스레드로 돌아간다 — 다른 채널이나 일반 폴백 채널로 가지 않는다.

이는 암묵적인 라우팅 가정에 의존하지 않고
결과 봉투 내 `ReplyChannelRef`로 강제된다.

## 오케스트레이션과의 관계

게이트웨이는 채널 인그레스와 에이전트/워크플로우 런타임 사이에 위치한다.
오케스트레이터를 대체하지 않는다 — 오케스트레이터를 호출할지 여부를 결정한다.

오케스트레이터는 다중 턴 에이전트 루프, 페이즈 워크플로우, 태스크 추적을 처리한다.
게이트웨이는 오케스트레이터가 호출되기 전에 어떤 경로를 열지 결정한다.
