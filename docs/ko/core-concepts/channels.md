# 채널

채널은 SoulFlow가 외부와 소통하는 입출력 레이어입니다. 메시지를 수신하고, 에이전트 결과를 응답으로 전송합니다.

## 지원 채널

| 채널 | 수신 방식 | 특이사항 |
|------|----------|----------|
| Slack | Socket Mode (WebSocket) | 스레드 응답, 파일 업로드 지원 |
| Telegram | Long Polling | 그룹/개인 채팅, 파일 첨부 |
| Discord | WebSocket Gateway | 서버/DM, 임베드 메시지 |
| Web Chat | SSE (Server-Sent Events) | 대시보드 내 브라우저 채팅 |

## 응답 방식

### 스트리밍

에이전트가 응답을 생성하는 동안 부분 결과를 주기적으로 전송합니다. 긴 작업에서 사용자가 진행 상황을 볼 수 있습니다.

대시보드 → **Settings** → `channel.streaming` 섹션에서 활성화/비활성화 및 플러시 주기를 조정할 수 있습니다.

### 메시지 그룹핑

짧은 시간 내 여러 응답을 하나로 묶어 전송합니다. 채팅 채널의 스팸을 방지합니다.

대시보드 → **Settings** → `channel.grouping` 섹션에서 설정합니다.

## 메시지 처리 파이프라인

```
수신 메시지
  1. 민감정보 자동 Sealing  ← 토큰, 패스워드 등을 Vault 참조로 대체
  2. 슬래시 커맨드 감지     ← /stop, /status, /secret 등
  3. 오케스트레이터 전달    ← 에이전트 실행
  4. 응답 디스패치          ← 스트리밍 or 단일 전송
```

## 동시 처리

여러 채널에서 동시에 메시지가 들어와도 독립적으로 처리됩니다.

동시 처리 수, 폴링 주기, 폴 당 최대 메시지 수는 대시보드 → **Settings** → `channel` 섹션에서 조정합니다.

## 전송 실패와 재시도

전송에 실패하면 지수 백오프로 재시도하고, 최대 재시도 횟수 초과 시 DLQ(Dead Letter Queue)에 보관합니다.

재시도 횟수, 백오프 주기, DLQ 활성화는 대시보드 → **Settings** → `channel.dispatch` 섹션에서 조정합니다.

`runtime/dlq/dlq.db`에서 실패 메시지를 확인하고 수동 재처리할 수 있습니다.

## 다중 인스턴스

대시보드 → **Channels** 페이지에서 같은 프로바이더의 여러 인스턴스를 추가할 수 있습니다. 예: Slack 채널 2개, Telegram 그룹 1개를 동시에 운영.

각 인스턴스는 독립적인 `instance_id`로 식별되며, 메시지 라우팅은 `instance_id` 기준으로 동작합니다.
